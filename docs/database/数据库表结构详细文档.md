# E-Tour 智能旅游攻略系统数据库表结构详细文档

## 概述

E-Tour 系统数据库包含 **14个核心数据表**，涵盖用户管理、攻略管理、行程管理、基础数据等模块。数据库采用 MySQL 8.0，字符集为 utf8mb4，支持完整的 Unicode 字符和表情符号。

## 数据表总览

| 模块 | 表名 | 中文名 | 功能描述 | 记录数 |
|------|------|--------|----------|--------|
| 用户管理 | user | 用户表 | 存储用户基本信息 | 1 |
| 用户管理 | member | 会员表 | 用户会员等级信息 | 0 |
| 用户管理 | user_preference | 用户偏好表 | 用户个性化偏好设置 | 0 |
| 用户管理 | order | 订单表 | 用户购买会员订单 | 0 |
| 攻略管理 | strategy | 攻略表 | 用户发布的旅游攻略 | 0 |
| 攻略管理 | strategy_tag | 攻略标签表 | 攻略分类标签 | 8 |
| 攻略管理 | strategy_tag_relation | 攻略标签关联表 | 攻略与标签的多对多关系 | 0 |
| 攻略管理 | comment | 评论表 | 攻略评论系统 | 0 |
| 攻略管理 | collect | 收藏表 | 用户收藏的攻略 | 0 |
| 攻略管理 | strategy_reject_reason | 攻略驳回原因表 | 管理员驳回攻略的原因记录 | 0 |

| 基础数据 | tags | 通用标签表 | 系统通用标签管理 | 0 |

## 核心实体关系分析

### 1. 用户模块关系（1:1, 1:N）
- **user ──1:1── member**：一个用户对应一个会员记录
- **user ──1:N── user_preference**：一个用户可以有多个偏好
- **user ──1:N── order**：一个用户可以创建多个订单
- **user ──1:N── strategy**：一个用户可以发布多个攻略
- **user ──1:N── comment**：一个用户可以发表多个评论
- **user ──1:N── collect**：一个用户可以收藏多个攻略
- **user ──1:N── trip**：一个用户可以创建多个行程

### 2. 攻略模块关系（N:M）
- **strategy ──N:M── strategy_tag**：多对多关系，通过strategy_tag_relation表关联
- **strategy ──1:N── strategy_reject_reason**：一个攻略可以有多个驳回记录
- **strategy ──1:N── comment**：一个攻略可以有多个评论
- **strategy ──1:N── collect**：一个攻略可以被多个用户收藏

### 3. 行程模块关系（N:M）
- **trip ──N:M── spot**：多对多关系，通过trip_spot表关联

## 索引设计策略

### 主键索引
- 所有表的id字段均设置为主键索引

### 唯一索引
- user.username：避免用户名重复
- member.user_id：一个用户仅对应一条会员记录
- order.order_no：订单编号唯一
- strategy_tag.name：标签名称唯一
- tags.name：通用标签名称唯一
- uk_strategy_tag：避免攻略重复关联同一标签
- uk_user_strategy：避免用户重复收藏同一攻略
- uk_trip_spot：避免行程重复添加同一景点

### 外键索引
- 所有外键字段均建立索引，提高关联查询效率

### 全文索引
- ft_strategy_title_content：支持攻略标题和内容的全文检索
- ft_spot_name_intro：支持景点名称和介绍的全文检索

### 普通索引
- 状态字段：status
- 时间字段：create_time, update_time
- 分类字段：destination, city, level
- 统计字段：rating

## 数据完整性约束

### 外键约束
- 所有外键关系均设置级联删除（ON DELETE CASCADE）
- 更新操作限制（ON UPDATE RESTRICT）

### 默认值约束
- 时间字段默认值为CURRENT_TIMESTAMP
- 计数字段默认值为0
- 状态字段默认值为正常状态

### 检查约束
- 通过应用程序层实现业务逻辑检查
- 数据库层面通过数据类型和长度限制

## 系统特点

1. **用户为中心**：所有业务数据均围绕user表展开
2. **多对多关系**：通过关联表实现灵活的标签和景点管理
3. **审核机制**：攻略发布需要管理员审核
4. **统计功能**：浏览量、点赞数、收藏数等统计字段
5. **AI集成**：行程支持AI优化功能
6. **全文检索**：支持攻略和景点的全文搜索
7. **地图集成**：景点支持经纬度定位
